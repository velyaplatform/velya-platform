# Anti-Goals Manifest

**Velya Hospital AI Platform**
**Document Type:** Formal Architecture Anti-Goals
**Date:** 2026-04-08
**Classification:** Internal — Engineering & Governance
**Status:** Active — Canonical Reference

---

## Purpose

Anti-goals define what Velya must never become. They are not aspirational — they are boundaries. Unlike requirements (which describe what to build), anti-goals describe drift patterns: the slow transformation of a system from what it was intended to be into something harmful.

Each anti-goal in this document has been chosen because it represents a real drift risk — a pattern that emerges naturally from common incentives (adding features, demonstrating capability, avoiding difficult conversations) and must be actively resisted.

Any team member who observes warning signs of an anti-goal drift should treat it as a production incident, not a design discussion.

## AG-001 — EHR Monolith Disguised as Platform

| Field                   | Value                                                                                                                                                                                                                                                                                                                                                                                       |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Anti-Goal ID**        | AG-001                                                                                                                                                                                                                                                                                                                                                                                      |
| **Name**                | EHR Monolith Disguised as Platform                                                                                                                                                                                                                                                                                                                                                          |
| **Description**         | Velya becomes a system where clinical workflows are hardcoded into service logic, configuration is inaccessible to non-engineers, and adding a new workflow type requires a code deployment. The platform's flexibility is architectural in documentation but rigid in practice.                                                                                                            |
| **Why It's Forbidden**  | Hospital operations vary by ward, shift, season, and patient population. A hardcoded workflow that works for a general medical ward fails in an ICU. A system that requires engineer involvement to change a discharge checklist is not a hospital operations platform — it is a custom EHR. The entire value proposition of an AI-native platform is configurability without code changes. |
| **Warning Signs**       | Service methods named after specific clinical workflows (e.g., `handleCardiacDischarge()`). Product backlog items requiring a deployment to change a clinical rule. Clinical staff requests that routinely become engineering tickets for logic changes rather than configuration changes. Runbooks that describe "call the dev team to change this setting."                               |
| **Detection Mechanism** | Quarterly architecture review: count the number of clinical workflow behaviors that require a code deployment vs. configuration change. Target: > 80% configurable without code.                                                                                                                                                                                                            |
| **Intervention**        | Extract hardcoded clinical logic into policy-engine rules. Create a workflow configuration interface accessible to authorized clinical operations staff. Refactor to event-driven, rule-driven architecture before the hardcoded surface area exceeds 20% of workflows.                                                                                                                     |

## AG-002 — Dashboard Theater

| Field                   | Value                                                                                                                                                                                                                                                                                                                                                                                                  |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Anti-Goal ID**        | AG-002                                                                                                                                                                                                                                                                                                                                                                                                 |
| **Name**                | Dashboard Theater                                                                                                                                                                                                                                                                                                                                                                                      |
| **Description**         | Velya's Grafana dashboards are impressive, comprehensive, and beautiful. They show system health, clinical throughput, agent performance, and cost metrics. They are viewed during demos and in standups. They do not drive clinical action. Clinicians do not use them. Alerts from the dashboard do not change what anyone does. The dashboards provide a feeling of control without actual control. |
| **Why It's Forbidden**  | Dashboard theater is expensive (engineering time to build and maintain) and dangerous (false confidence that you know what's happening). In a hospital context, a dashboard that shows "all green" when clinical staff are overwhelmed and tasks are delayed is not just useless — it is actively harmful because it discourages investigation.                                                        |
| **Warning Signs**       | Dashboard views primarily spike during scheduled reviews, not during incidents. No Grafana dashboard has ever triggered a human action outside of a planned review. Clinicians cannot name which dashboard they use for daily operations. Dashboards have not been updated in > 60 days. An incident occurs that the dashboards did not surface.                                                       |
| **Detection Mechanism** | Track dashboard view events (Grafana usage analytics). Measure the ratio of dashboard-triggered actions to total incidents. If < 20% of incidents were surfaced by dashboards before being reported by humans, the dashboards are decorative.                                                                                                                                                          |
| **Intervention**        | Deprecate dashboards that have not driven an action in 90 days. Convert meaningful dashboard panels into alerting rules. Require every new dashboard to have a documented "action threshold" — the condition at which a human acts on what they see.                                                                                                                                                   |

## AG-003 — Automation Without Accountability

| Field                   | Value                                                                                                                                                                                                                                                                                                                                                            |
| ----------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Anti-Goal ID**        | AG-003                                                                                                                                                                                                                                                                                                                                                           |
| **Name**                | Automation Without Accountability                                                                                                                                                                                                                                                                                                                                |
| **Description**         | Tasks are completed by agents, workflows execute, records are updated, and notifications are sent — but no human can reconstruct who or what made a specific decision, why, and on what data. The automation is efficient but opaque. When a clinical error occurs, the trail goes cold at "the system did it."                                                  |
| **Why It's Forbidden**  | HIPAA requires auditability of all actions on PHI (45 CFR §164.312(b)). Clinical governance requires that every clinical decision be attributable to a human or a validated automated system. "The agent did it" is not an acceptable answer in a regulatory investigation, a malpractice proceeding, or a root cause analysis after a clinical incident.        |
| **Warning Signs**       | decision-log-service is not queried during incident investigations. Audit log records show generic action types without agent identity or reasoning. Clinical staff describe agent actions as "the system just does things." No human can explain why a specific task was created or a recommendation was made without asking an engineer to dig into NATS logs. |
| **Detection Mechanism** | Monthly audit: randomly select 5 agent decisions from decision-log-service. Can they be fully reconstructed (who, what, why, on what data) in < 10 minutes by a non-engineer? If not, accountability is insufficient.                                                                                                                                            |
| **Intervention**        | Mandate that every agent action writes a decision record to decision-log-service before the action completes. Implement a "decision explorer" UI for clinical supervisors. Require that every audit-service record links to the full agent decision chain.                                                                                                       |

## AG-004 — AI With Implicit Authority

| Field                   | Value                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Anti-Goal ID**        | AG-004                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **Name**                | AI With Implicit Authority                                                                                                                                                                                                                                                                                                                                                                                                                    |
| **Description**         | AI recommendations become so routinely accepted that they are functionally treated as orders. Clinical staff acknowledge them without review. The confidence indicator stops being read. The "AI Suggestion" label is ignored. Effectively, an LLM is making clinical decisions with human rubber-stamping.                                                                                                                                   |
| **Why It's Forbidden**  | AI systems have failure modes that differ from human judgment: they hallucinate, they are overconfident, they fail silently on out-of-distribution cases. A clinician who is genuinely reviewing a recommendation can catch these failures. A clinician who is rubber-stamping cannot. In a hospital, the difference is patient harm. Regulatory frameworks (FDA, CMS) also require that clinical decisions remain under physician authority. |
| **Warning Signs**       | AI recommendation override rate drops below 10% (mathematical impossibility for an imperfect system). Clinicians complete AI reviews in < 15 seconds consistently. Clinical staff describe Velya as "the AI that makes decisions." Product roadmap includes "auto-approve recommendations with confidence > 0.95."                                                                                                                            |
| **Detection Mechanism** | Governance dashboard tracks per-clinician AI override rate. Statistical outliers (< 5% override rate over 30 days) are reviewed. AI recommendations made with incorrect supporting data are tracked to measure how often the error was caught.                                                                                                                                                                                                |
| **Intervention**        | UI controls enforce engagement (Control 4 in cognitive-safety-controls.md). Governance policy sets minimum override rate expectations. Periodic "AI in training mode" where recommendations require independent assessment before reveal. Any proposal to auto-approve AI recommendations without human review is blocked by governance vote.                                                                                                 |

## AG-005 — Happy Path Only Architecture

| Field                   | Value                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Anti-Goal ID**        | AG-005                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **Name**                | Happy Path Only Architecture                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| **Description**         | Velya works excellently when all services are running, network is stable, NATS delivers every message, the AI returns a confident recommendation, and the clinician is fresh and fully trained. Under any deviation from these conditions — service down, partial failure, degraded connectivity, edge-case patient data, fatigued user — the system behaves unpredictably, silently fails, or presents incorrect information without any indication that something is wrong. |
| **Why It's Forbidden**  | Hospitals operate in a world of constant degraded conditions: shift changes, maintenance windows, network issues, legacy equipment, atypical patients, overwhelmed staff. A clinical platform that fails silently or degrades without notification under normal hospital conditions is not a hospital platform — it is a demo.                                                                                                                                                |
| **Warning Signs**       | Error handling in services returns HTTP 500 with no structured error body. Tests only cover the success path. velya-web has no degraded mode rendering. NATS consumer failure is not tested. Disaster recovery has not been drilled. The phrase "that won't happen in production" appears in engineering discussions.                                                                                                                                                         |
| **Detection Mechanism** | Quarterly chaos engineering exercise: inject failures into 3 randomly selected components and measure platform behavior. Score: (1) how many failures resulted in silent degradation, (2) how many failures triggered correct user notification, (3) how quickly clinical staff understood the impact.                                                                                                                                                                        |
| **Intervention**        | Chaos engineering as a continuous practice, not a one-time event. Every service must define its degraded mode behavior before its first production deployment. Failure modes are a first-class architectural concern — a feature is not complete until its failure behavior is defined, implemented, and tested.                                                                                                                                                              |

## AG-006 — Single Person Dependency

| Field                   | Value                                                                                                                                                                                                                                                                                                                           |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Anti-Goal ID**        | AG-006                                                                                                                                                                                                                                                                                                                          |
| **Name**                | Single Person Dependency                                                                                                                                                                                                                                                                                                        |
| **Description**         | System knowledge — how to deploy, how to debug, how to change a configuration, how a specific agent works, why a specific architectural decision was made — is concentrated in one developer. The platform cannot be maintained, debugged, or evolved without that person's involvement.                                        |
| **Why It's Forbidden**  | A hospital platform with single-person dependency is a patient safety risk. When that person is unavailable (illness, vacation, resignation), the platform cannot be safely operated. Incidents cannot be investigated. Clinical staff cannot get support. This is not a staffing concern — it is a safety engineering concern. |
| **Warning Signs**       | Incidents require escalation to one specific person. Documentation of decisions is absent or informal. New team members cannot deploy the platform without pairing with the lead developer. Only one person has credentials to critical systems. Runbooks say "ask [name]."                                                     |
| **Detection Mechanism** | Bus factor test: can any two team members (excluding the most senior) deploy the full platform from scratch using only written documentation? If not, bus factor = 1.                                                                                                                                                           |
| **Intervention**        | Enforce documentation as a deployment prerequisite. Rotate on-call ownership. Require all credentials to be accessible via shared credential management. Every operational action must have a runbook. Cross-train on all critical system areas.                                                                                |

## AG-007 — Bureaucratic Automation Machine

| Field                   | Value                                                                                                                                                                                                                                                                                                                                                                                     |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Anti-Goal ID**        | AG-007                                                                                                                                                                                                                                                                                                                                                                                    |
| **Name**                | Bureaucratic Automation Machine                                                                                                                                                                                                                                                                                                                                                           |
| **Description**         | Agents create tasks, reports, checklists, and notifications faster than humans can process them. The agent governance layer generates more compliance reports than anyone reads. The task inbox is always full — not because of clinical load, but because agents are generating administrative tasks that provide no clinical value. The platform adds overhead rather than reducing it. |
| **Why It's Forbidden**  | The purpose of Velya's automation is to reduce clinical administrative burden. If the platform generates more administrative work than it eliminates, it has failed its core purpose. Clinical staff who spend more time managing Velya than caring for patients will reject the platform — and they would be right to.                                                                   |
| **Warning Signs**       | Clinicians describe Velya as "another thing to check." Task inbox items closed without being read exceeds 40%. Agent governance reports are not reviewed in recurring meetings. Task creation rate by agents consistently exceeds task completion rate. "We need an alert for the alerts."                                                                                                |
| **Detection Mechanism** | Track task-to-action ratio (tasks completed with recorded clinical action vs. tasks closed/dismissed). If > 30% of agent-created tasks are dismissed without action, the agent is generating noise.                                                                                                                                                                                       |
| **Intervention**        | Each agent must justify its output rate. Agents with high dismiss rates are quarantined for review. Agent scope is narrowed before output rate is increased. New agent capabilities require a clinical value demonstration before production deployment.                                                                                                                                  |

## AG-008 — Unexplainable Agent Runtime

| Field                   | Value                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Anti-Goal ID**        | AG-008                                                                                                                                                                                                                                                                                                                                                                                                                               |
| **Name**                | Unexplainable Agent Runtime                                                                                                                                                                                                                                                                                                                                                                                                          |
| **Description**         | Agents make decisions — routing tasks, generating recommendations, triggering workflows — but no one can trace the reasoning chain. The decision-log-service records outcomes but not the chain of context, tools, and model output that produced them. When a clinical supervisor asks "why did the agent recommend early discharge for this patient?", the answer is "the model produced that output" with no further information. |
| **Why It's Forbidden**  | Unexplainability is a clinical governance failure. Clinical decisions must be auditable. An agent whose reasoning cannot be reconstructed is functionally indistinguishable from a random decision process. It cannot be validated, improved, or corrected. Regulatory bodies will not accept "the AI decided" as a clinical justification.                                                                                          |
| **Warning Signs**       | decision-log-service stores only the final decision, not the reasoning chain. There is no "explain this recommendation" feature in velya-web. Prompt templates are not version-controlled. Context assembled for an agent call cannot be reconstructed from logs.                                                                                                                                                                    |
| **Detection Mechanism** | Monthly: randomly select one agent decision and attempt to reconstruct the full reasoning chain — input context, tools called, model output, post-processing. If reconstruction takes > 30 minutes or is incomplete, explainability is insufficient.                                                                                                                                                                                 |
| **Intervention**        | Log the full agent context (sanitized of PHI for storage, with reference IDs to recover the full data) alongside every decision. Version-control prompt templates. Implement a decision replay feature that can reconstruct what the agent saw. Build the "explain this" UI feature before production deployment.                                                                                                                    |

## AG-009 — Cosmetic Frontend

| Field                   | Value                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Anti-Goal ID**        | AG-009                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Name**                | Cosmetic Frontend                                                                                                                                                                                                                                                                                                                                                                                                                                |
| **Description**         | velya-web looks professional, modern, and well-designed. It demos well. Clinical staff who see it in a demo respond positively. But in actual ward use, it increases cognitive load rather than reducing it, misses critical alerts in visual noise, doesn't work on mobile in poor lighting, and fails to provide the right information at the right moment for the right role. The UI is designed for demo aesthetics, not clinical cognition. |
| **Why It's Forbidden**  | A frontend that looks good but performs poorly in clinical conditions is more dangerous than a plain one. It creates confidence that the system is well-designed when it isn't. Clinical staff who discover the gap between demo experience and ward experience will lose trust in the entire platform.                                                                                                                                          |
| **Warning Signs**       | UI design decisions are primarily driven by visual consistency and brand guidelines. There has been no usability testing with clinical staff in clinical conditions. Frontend features are demoed on a desktop in a quiet room. Feedback from clinical staff about the UI is positive in meetings and negative in hallways.                                                                                                                      |
| **Detection Mechanism** | Mandatory usability testing with clinical staff in ward conditions (or simulated ward conditions: mobile device, bright lighting, time pressure, multiple concurrent tasks) before every major release. At least one session must involve a shift-change handoff scenario.                                                                                                                                                                       |
| **Intervention**        | Clinical usability testing is a release gate, not a post-release improvement. Any UI change affecting clinical workflows must be validated by a clinical user. The human-factors-risk-model.md and cognitive-safety-controls.md are functional requirements, not design guidelines.                                                                                                                                                              |

## AG-010 — Self-Improving System Without Guardrails

| Field                   | Value                                                                                                                                                                                                                                                                                                                                                                 |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Anti-Goal ID**        | AG-010                                                                                                                                                                                                                                                                                                                                                                |
| **Name**                | Self-Improving System Without Guardrails                                                                                                                                                                                                                                                                                                                              |
| **Description**         | Agents accumulate memory, learn from feedback, update their own prompt templates, or modify their configuration based on observed outcomes — without requiring human review and validation of the changes. The system improves itself in ways that are not reviewable, not reversible, and not auditable.                                                             |
| **Why It's Forbidden**  | A self-modifying clinical AI system that evolves without human oversight cannot be validated. Changes that appear to improve performance on measured metrics may degrade performance on unmeasured clinical safety metrics. The validation regime that approved the original system does not apply to the modified system. This is an open-ended patient safety risk. |
| **Warning Signs**       | memory-service entries are written without requiring human review. Agent configuration is modified by the agent itself. Prompt templates are updated based on agent self-assessment. Learning propagation occurs without a defined review step. "The agent got better at this" cannot be traced to a specific approved change.                                        |
| **Detection Mechanism** | Any modification to agent configuration, prompt templates, memory-service entries, or policy rules must generate an audit record with the proposed change, the origin agent, and the approving human. If any change record shows no human approver, it is a violation.                                                                                                |
| **Intervention**        | All agent self-improvement pathways are mediated by a human approval step. Memory writes from agents are quarantined pending review for high-impact memory types. Prompt template changes require governance approval before deployment. Kill switch for learning propagation (see kill-switch-matrix.md).                                                            |

## AG-011 — Feature Accumulation Without Retirement

| Field                   | Value                                                                                                                                                                                                                                                                                                                                                                                      |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Anti-Goal ID**        | AG-011                                                                                                                                                                                                                                                                                                                                                                                     |
| **Name**                | Feature Accumulation Without Retirement                                                                                                                                                                                                                                                                                                                                                    |
| **Description**         | Every capability added to Velya stays in the platform forever. Deprecated agents continue to run. Unused Grafana dashboards continue to be maintained. Features that turned out to provide no clinical value remain in the UI, adding visual noise. Services accumulate technical debt because no one is authorized to remove anything. The platform grows in complexity but not in value. |
| **Why It's Forbidden**  | Complexity is a clinical safety risk. Every additional code path, every additional agent, every additional dashboard is a surface for failure and a tax on engineering attention. Unused features create false confidence (if it's deployed, someone must be using it). Features that no clinician uses are hidden cognitive overhead that clinical staff must mentally navigate around.   |
| **Warning Signs**       | Feature usage metrics are not tracked. Pull requests adding features are merged without an associated deprecation plan. The agent registry contains agents with no recorded invocations in > 30 days. velya-web navigation has items that have never been clicked in the last 90 days according to analytics.                                                                              |
| **Detection Mechanism** | Quarterly feature usage review. Any feature with < 5% of expected usage over 90 days is placed on the deprecation track. Deprecation requires a 30-day notice period and confirmation that no clinical workflow depends on the feature.                                                                                                                                                    |
| **Intervention**        | Feature retirement is a product process, not an engineering afterthought. Every feature has an owner. Features without owners are automatically placed in the deprecation track. Removing a feature requires the same review as adding one.                                                                                                                                                |

## AG-012 — Metrics Theater

| Field                   | Value                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Anti-Goal ID**        | AG-012                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| **Name**                | Metrics Theater                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| **Description**         | Velya collects extensive metrics: agent invocation rates, task completion rates, AI recommendation acceptance rates, clinical throughput, cost per inference. These metrics appear in dashboards and are referenced in stakeholder reports. They are not used to make decisions. Threshold alerts don't trigger action. The metrics change over time and no one investigates why. Decisions are made by intuition while metrics provide post-hoc justification. |
| **Why It's Forbidden**  | Metrics theater is expensive (instrumentation, storage, dashboard maintenance) and dangerous (creates the false impression that the system is understood). In a clinical platform, unreviewed metrics mean clinical safety signals may be visible in the data but not acted upon.                                                                                                                                                                               |
| **Warning Signs**       | The last time a metric change triggered an engineering investigation was > 60 days ago. Dashboards show trend lines that clearly indicate a change, and no one has commented on them. Metrics are added to dashboards but never removed. There is no defined "metric owner" for any panel.                                                                                                                                                                      |
| **Detection Mechanism** | Each metric must have a defined response action. Monthly: for each active Prometheus metric, confirm that at least one alert or review action is triggered by a threshold. Metrics with no associated alert and no evidence of human review are candidates for removal.                                                                                                                                                                                         |
| **Intervention**        | Implement a "metrics review" as part of each sprint retro. Each metric must be tied to a decision or removed. Alert rules must exist for every metric with a defined safety significance. Dashboard panels must include a "last reviewed" annotation.                                                                                                                                                                                                           |

## AG-013 — Shadow Mode Theater

| Field                   | Value                                                                                                                                                                                                                                                                                                                                                                                                           |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Anti-Goal ID**        | AG-013                                                                                                                                                                                                                                                                                                                                                                                                          |
| **Name**                | Shadow Mode Theater                                                                                                                                                                                                                                                                                                                                                                                             |
| **Description**         | Agents and AI features are deployed in "shadow mode" indefinitely. They observe and produce outputs that are logged but never acted upon. Shadow mode is used as a permanent state to avoid accountability — the system generates recommendations, but since they're "just shadow mode," no one reviews them, no one validates them, and no one makes a decision about whether to graduate them to active mode. |
| **Why It's Forbidden**  | Shadow mode has a defined purpose: validate that a new capability behaves correctly before granting it clinical authority. If shadow mode becomes a permanent state, it provides no clinical value while consuming engineering resources and AI compute budget. More dangerously, it creates the impression of an active AI system ("Velya's AI is running") while actually delivering nothing.                 |
| **Warning Signs**       | Any agent or feature has been in shadow mode for > 90 days without a documented graduation plan. Shadow mode outputs are not reviewed as part of any recurring process. The phrase "we'll figure out when to go live later" appears in the backlog. Shadow mode is cited as evidence of AI capability in stakeholder communications.                                                                            |
| **Detection Mechanism** | Every shadow mode deployment has a defined graduation criteria document and a maximum shadow duration. At 60 days, an automatic review is triggered. At 90 days, shadow mode is automatically suspended unless explicitly renewed with governance approval.                                                                                                                                                     |
| **Intervention**        | Shadow mode graduation criteria are defined before shadow mode deployment begins, not after. Shadow mode outputs are reviewed weekly by a defined reviewer. If graduation criteria cannot be defined, shadow mode should not begin.                                                                                                                                                                             |

## AG-014 — Governance Overhead Without Governance Value

| Field                   | Value                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Anti-Goal ID**        | AG-014                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| **Name**                | Governance Overhead Without Governance Value                                                                                                                                                                                                                                                                                                                                                                                                            |
| **Description**         | The Velya governance model produces reports: agent scorecards, policy compliance reports, audit summaries, risk registers. These reports are generated by agents, formatted by agents, and archived by agents. They are referenced in governance meetings as evidence of governance. They are not read by anyone who makes clinical decisions. The governance process exists to satisfy itself, not to protect patients or improve clinical operations. |
| **Why It's Forbidden**  | Governance that doesn't change behavior is bureaucracy. Bureaucracy consumes engineering and clinical time and creates false confidence that risks are managed when they aren't. In a hospital platform, governance theater is a direct patient safety risk because it consumes the attention and resources that should be directed at actual safety controls.                                                                                          |
| **Warning Signs**       | Governance reports are generated but not discussed. Risk register items are never closed. Policy violations are logged but not investigated. Agent scorecards show declining performance, and no action is taken. The governance meeting agenda is "review reports" rather than "make decisions."                                                                                                                                                       |
| **Detection Mechanism** | For each governance output (report, scorecard, register), track: who read it, what decision it informed, what action resulted. If a governance artifact has no associated action in 60 days, it is either providing no value or the process is broken.                                                                                                                                                                                                  |
| **Intervention**        | Governance outputs are decision inputs, not archives. Each report must have a named recipient and a named decision it informs. If the decision is already made without the report, the report is unnecessary. Reduce governance reporting frequency and increase governance action frequency.                                                                                                                                                           |

## AG-015 — Vendor Lock-In Disguised as Architecture

| Field                   | Value                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Anti-Goal ID**        | AG-015                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **Name**                | Vendor Lock-In Disguised as Architecture                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Description**         | Velya's architecture is so tightly coupled to specific vendor offerings — Anthropic's Claude API, AWS-specific services (SQS, DynamoDB, EventBridge via LocalStack), ArgoCD-specific GitOps patterns, NATS JetStream — that migrating away from any of them requires a platform rewrite. The abstraction layer (ai-gateway, event bus interfaces) that was intended to prevent this becomes a thin wrapper that provides an abstraction illusion while actually encoding vendor-specific behavior. |
| **Why It's Forbidden**  | Hospital procurement relationships change. Vendors change pricing, terms, or capabilities. A platform that cannot be migrated is a platform that cannot be independently operated or negotiated with. In a clinical setting, inability to migrate also means inability to adopt superior clinical AI models as they emerge. Lock-in reduces clinical quality by constraining model selection.                                                                                                      |
| **Warning Signs**       | ai-gateway has Claude-specific response handling code that cannot be swapped for another provider. NATS-specific message headers are used in clinical business logic rather than in the messaging layer. AWS SDK calls appear in NestJS services rather than being behind an abstraction. ArgoCD Application-specific annotations are embedded in application manifests. LocalStack-specific error codes appear in application error handling.                                                     |
| **Detection Mechanism** | Annual portability review: for each major vendor dependency (Anthropic, AWS, NATS, ArgoCD), estimate the migration effort to a functional alternative. If any single migration would require > 3 months of engineering work, lock-in is excessive.                                                                                                                                                                                                                                                 |
| **Intervention**        | ai-gateway must implement a provider interface that can serve multiple LLM providers. All cloud-specific calls go through an infrastructure abstraction layer (not directly via SDK). Event bus subjects and schemas are defined independently of NATS specifics. Portability tests (can we run this against a different provider?) are added to the CI pipeline.                                                                                                                                  |
